Raytrace

January 5, 2021

The raytracer challenge book provides a test driven, easy to to get started with, from scratch, step by step project to build yourself a raytracer to create pretty pictures.

Turns out a raytracer makes for a nice benchmark and testcase for language implementations, too.

Introduction

Raytracing is a compute intensive process, and scaling up image sizes or scene complexity can result in long run times - motivating optimisation to reduce waiting time!

The built-in screenshot facility of Ubuntu helps.

Multi-threading

CPU’s have multiple cores we can use - even the Pi400 has 4 ARM cores. Raytracing is easily made parallel, so we can hope for a linear speedup.

4 cores 1 core
9.85s 29.4s

Not 4x faster, but 3x faster. Would seem to indicate a factor other than CPU availability impacting performance.

Stack allocation vs Heap allocation

How data is represented in the ray tracer matters, too: data such as a point, vector, or colour, are tuples of scalar values. Some programming languages permit stack allocation of such aggregate values, and others require allocation on a heap area, potentially with garbage collection.

In C#, the two alternatives are struct and class, respectively; in Oberon, they are RECORD and POINTER TO RECORD.

We can decide that a variable created via a special operator or function e.g. new is heap allocated, or just make that inherent in type as here using struct for stack allocation or record for heap allocation.

type T* : struct 
    x*, y*, z*, w : FLOAT;
end

type Colour* : struct r,g,b : FLOAT end

vs

type T* : record 
    x*, y*, z*, w : FLOAT;
end

type Colour* : record r,g,b : FLOAT end

The impact of heap allocation is two-fold: Firstly, multiple threads may attempt to allocate data on a (shared) heap at the same time: this may require synchronisation via e.g. mutex to prevent memory corruption of the heap layout. The contention for a mutex can result in a core waiting for another, while not progressing with useful work. Secondly, the data allocated may be “live” for only a short time, before becoming garbage that requires to be collected to prevent the heap area being exhausted.

Stack allocation GC Heap allocation
2.16s 6.03s

Compiler optimisations!

Can we just make a compiler that is really good at optimizing code? Well, maybe. It may be intractable in general, and even in the specific, it’s a lot of work.

We can cheat though - if the language we are writing in compiles to another programming language (e.g. C) for which an optimising compiler is available, we can “just” use the optimisations provided by that compiler.

Opts (gcc -O3) No opts (gcc -O0) Opts (clang -O3) No opts (clang -O0)
2.16s 9.85s 2.46s 10.8s

So, unsurprisingly, compiler optimisations make code faster. In this case, for this application, on ARM64 Linux (Pi400), gcc appears to be slightly faster.

Caching inverse matrixes

The ray tracer challenge raytracer design frequently computes and recomputes the inverse of matrixes; the suggestion on the forum is to cache the computed inverse and therefore reduce the computation required.

Caching Non-caching
2.16s 97.24s

The speedup given by avoiding uncessary recomputation is large!

Conclusions?

No conclusions, really: except that it is worth using application specific optimisations, and generic optimisations (parllelism, data layout/allocation) in addition to compiler optimisations.

Note: all measurements here are thoroughly unscientific, this is not any attempt to be rigourous or precise.
